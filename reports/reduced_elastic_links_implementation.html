What Changed

Public API

  • JointType.ELASTIC owns a floating rigid frame plus reduced modal coordinates.
  • ModelBuilder.add_link_elastic() creates an elastic link for articulated mechanisms.
  • ModelBuilder.add_body_elastic() mirrors add_body() for stand-alone floating elastic bodies.
  • ModelBuilder.add_joint_elastic() is the state-owner joint used by those helpers.
  • ModalBasis stores sampled linear mode values; ModalGeneratorBeam, ModalGeneratorPOD, and ModalGeneratorSampled construct reusable bases.
  • The viewer now deforms elastic shape meshes from sampled modal basis values during log_state().

State Layout

  • joint_q: [px, py, pz, qx, qy, qz, qw, eta0, eta1, ...].
  • joint_qd: [vx, vy, vz, wx, wy, wz, eta_dot0, eta_dot1, ...].
  • The body still has a normal body_q and body_qd slot for rigid-body infrastructure.
  • Modal endpoint samples are cached from the existing joint parent/child transforms, with no separate attachment API.
  • Multiple elastic bodies can share one modal basis through elastic_basis; each joint/render sample keeps its basis-local sample index.

Solver Pipeline

Stage Implementation
Frame sync The elastic owner joint's first 7/6 coordinates are copied into the normal body pose/twist slots before rigid VBD runs.
Modal prediction Each modal coordinate uses an implicit Euler update with modal mass, stiffness, damping, and Control.joint_f.
Attachment evaluation Rigid VBD evaluates ordinary joint anchors through sampled mode shapes when the parent or child body is elastic.
Modal constraint solve After each rigid VBD pass, modal coordinates receive a diagonal implicit solve from joint-anchor residual gradients projected through the mode samples.
Frame write-back The solved body pose/twist is copied back into the owner joint, preserving the solved modal coordinates.
Viewer draw Builder finalization samples elastic shape vertices and a diagnostic centerline. ViewerGL hides the rigid instance for those shapes and uploads deformed mesh vertices each frame.

Mode Basis

The demo modes are now generated through ModalBasis rather than per-body Python callables. ModalBasis stores GPU-friendly samples phi[sample, mode]; builders add attachment and surface sample points locally to that basis, then flatten the values into the current VBD arrays.

The beam generator produces standard small-deflection fields. The four-bar coupler uses one axial extension mode and the first two pinned-pinned Euler-Bernoulli bending modes. The bending displacement includes the plane-section axial term u_x = -y dw/dx, so the rendered beam cross-section rotates consistently with the neutral-axis deflection.

The revolute endpoint fixture uses five polynomial torsion modes about the link axis, u = theta(x) cross r. The shaft surface vertices are supplied directly as modal samples, so each displayed vertex uses analytic torsion phi values rather than inverse-distance interpolation from a coarser basis. The initial modal coordinates use the constant-strain linear torsion mode for a smooth 90 degree tip twist, then the shaft is released with no ongoing modal forcing.

The static cantilever example uses the exact Euler-Bernoulli tip-load shape with analytic static tip displacement delta = P L^3 / (3 E I). The vibration example uses a stiffer, lightly damped finite-mass mode and releases the beam from an initial deflection, producing a quick underdamped ring-down. ModalGeneratorPOD starts the exemplar-deformation path by extracting linear modes from sampled displacement snapshots.

Current Scope

  • Endpoint, solver, and viewer modes are translational FFRF samples. Mesh rendering now evaluates the same basis at each render vertex.
  • The modal constraint solve is diagonal per mode. This keeps the implementation small and robust for few-mode links, but dense modal coupling is still future work.
  • Physical modal solving is implemented for VBD. Other solvers can carry the state layout but do not yet provide the same elastic-mode constraint solve.
  • Shape deformation is currently a viewer path; collision shapes are still handled by the rigid-body collision pipeline.
  • Existing parent_xform and child_xform arguments remain the attachment API for now.
