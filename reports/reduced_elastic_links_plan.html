Reduced Elastic Links in Newton

Plan for adding a link type that owns a rigid floating frame plus reduced elastic coordinates, participates in the body state arrays, and can be constrained to rigid links through standard joints such as revolute joints.

Summary

The cleanest Newton integration is to model a reduced elastic link as a body-owned state block: one body slot in body_q/body_qd plus one state-owner joint whose generalized coordinates are [pos(3), quat(4), mode_0, mode_1, ...] and velocities are [linear(3), angular(3), mode_dot_0, mode_dot_1, ...]. Physical connections remain normal Newton joints, but their anchors can be attached to deformed material points on the elastic body.

What should be new

  • A reduced elastic link API, proposed as ModelBuilder.add_link_elastic() plus a free-body wrapper add_body_elastic().
  • A new state-owner joint type, proposed as JointType.ELASTIC, with 7 + N coordinates and 6 + N velocities.
  • Mode data stored on the model: modal mass, stiffness, damping, shape functions sampled at joint endpoint transforms, and optional inertia shape integrals.
  • Elastic-aware joint endpoint evaluation so a revolute joint can connect to a deformed material frame without adding public attachment IDs.
  • A VBD block solve for 6 + N unknowns, where the first six unknowns are floating-frame motion and the remaining unknowns are modal amplitudes.

What should stay stable

  • Rigid bodies and existing joints keep their public behavior.
  • Existing body_q/body_qd remains the pose and spatial velocity source for collision, rendering, and body-space shapes.
  • Existing revolute/ball/fixed/D6 joint solvers are reused after replacing rigid-only endpoint evaluation with endpoint evaluators that understand elastic endpoints.
  • VBD is the first solver target. Generalized solvers can be supported later once the reduced elastic body block is validated.

The physical revolute joints use their existing parent/child transforms as material frames on rigid or elastic bodies. The elastic link's state-owner joint stores the floating frame and modal coordinates.

FFRF Model

The floating frame of reference formulation separates large rigid motion from small local deformation. Shabana's treatment frames this as a deformable body with a moving reference frame, a linear elastic displacement field in that frame, and inertia coupling terms that depend on the assumed displacement field.

State

q_elastic  = [r_x, r_y, r_z, quat_x, quat_y, quat_z, quat_w, eta_0, ..., eta_N-1]
qd_elastic = [v_x, v_y, v_z, omega_x, omega_y, omega_z, eta_dot_0, ..., eta_dot_N-1]

For a material point with local rest position x0, local mode basis Phi(x0), modal amplitudes eta, world translation r, and world rotation A from the floating frame: 

u_local(x0, eta) = x0 + Phi(x0) * eta
x_world          = r + A * u_local
v_world          = v + omega x (A * u_local) + A * Phi(x0) * eta_dot

A strict FFRF implementation also needs generalized mass terms and inertia shape integrals to capture coupling between reference motion and elastic deformation. The first implementation can accept diagonal modal mass for minimal tests, but the model storage should be designed for dense modal mass and coupling data so the public API does not dead-end.

Quantity Initial storage Why it matters
Mode amplitudes joint_q[q_start + 7 : q_start + 7 + N] Matches the requested joint-array layout and keeps reduced elastic coordinates in Newton's generalized state.
Mode velocities joint_qd[qd_start + 6 : qd_start + 6 + N] Enables damping, modal kinetic energy, and finite-difference validation.
Floating pose joint_q[q_start : q_start + 7] and body_q[body] Preserves the existing body pipeline for collision and rendering.
Endpoint mode values Internal mode samples generated from joint_X_p / joint_X_c Lets ordinary joints constrain deformed endpoint position and, when available, deformed endpoint orientation.
Elastic energy elastic_mode_stiffness, elastic_mode_damping, elastic_mode_mass Provides the modal oscillator terms for VBD's implicit objective.

Current Newton Pipeline

The current code already has the two state spaces needed for this feature: generalized joint arrays and body arrays. The missing piece is a variable-size elastic state owner and endpoint evaluation that maps modal deformation into joint constraints.

File Relevant behavior Implication
newton/_src/sim/enums.py:41 JointType.dof_count() is type/axis based and assumes fixed-size joint coordinate counts. JointType.ELASTIC needs per-joint mode count support, not only enum-global sizing.
newton/_src/sim/builder.py:3427 add_joint() appends type, parent/child, anchors, DOF dimensions, starts, and per-DOF arrays. The builder should get an elastic-specific path that appends 7 + N coords, 6 + N velocities, and modal arrays coherently.
newton/_src/sim/articulation.py FK/IK branches on joint type and writes body_q/body_qd from joint_q/joint_qd. An elastic owner needs FK sync for the floating frame and mode coordinates, even if VBD is the first dynamic target.
newton/_src/solvers/vbd/solver_vbd.py VBD treats rigid bodies as maximal-coordinate blocks and supports rigid joint constraints; FREE joints have no solver constraints. The elastic owner should behave like a zero-constraint state owner, while physical joints remain solver constraints.
newton/_src/solvers/vbd/rigid_vbd_kernels.py Rigid joint force/Hessian kernels assume anchors are fixed transforms on rigid body frames and solve a 6 DOF body block. Replace rigid-only anchor evaluation with an endpoint evaluator and add a 6 + N elastic body block solve.
newton/tests/test_physics_validation.py Four-bar mechanisms already exist, and VBD paths should use loop joints rather than equality constraints. The elastic four-bar regression should be built as rigid joints plus loop-closing constraints supported by VBD.

Current VBD updates body_q/body_qd as primary state for rigid bodies. It does not generally integrate joint positions for rigid joints. Reduced elastic links therefore need explicit synchronization in both directions: initialize the body slot from the owner joint, solve the elastic block, then write the solved pose and modal state back to joint_q/joint_qd.

Architecture Proposal

State-owner joint plus physical joints

Use a new zero-constraint state-owner joint for each reduced elastic body. This joint owns the requested generalized coordinates. Physical connectivity remains represented by ordinary joints between body IDs. If either endpoint body is elastic, the existing parent_xform or child_xform on that joint is interpreted as a material frame in the elastic body's floating-frame coordinates. 

# Proposed user-facing shape.
builder = newton.ModelBuilder()

ground = -1
left = builder.add_link(...)
coupler = builder.add_link_elastic(
    parent=-1,
    xform=wp.transform(...),
    modes=builder.ElasticModes(
        mass=[...],
        stiffness=[...],
        damping=[...],
    ),
)
right = builder.add_link(...)

builder.add_joint_revolute(
    parent=left,
    child=coupler,
    parent_xform=wp.transform(...),
    child_xform=wp.transform(...),  # material frame on the elastic coupler
    axis=...,
)
builder.add_joint_revolute(
    parent=coupler,
    child=right,
    parent_xform=wp.transform(...),  # material frame on the elastic coupler
    child_xform=wp.transform(...),
    axis=...,
)

This keeps the existing joint-builder surface. The public docs and examples should use public Newton modules only; internal implementation details stay in newton/_src.

elastic_endpoint_joint, elastic_endpoint_side [elastic_endpoint_count] Maps each internal elastic endpoint back to a joint and parent/child side.

elastic_endpoint_psi Optional flat [endpoint, mode, xyz] storage Rotational/slope mode values sampled at the same joint endpoint transform.
joint_mode_count [joint_count] Needed because JointType.ELASTIC has per-joint size 7 + N / 6 + N.

Builder behavior

  1. add_link_elastic() calls the existing body allocation path to create a normal body slot.
  2. It appends a JointType.ELASTIC owner joint with parent -1 by default, no solver constraint slots, and a child equal to the elastic body.
  3. It writes the first 7 coordinates from the body pose, initializes mode amplitudes to zero or user input, and initializes the first 6 velocities from the body spatial velocity.
  4. It appends modal properties and records the body-to-elastic and elastic-to-joint maps.
  5. Existing joint builders keep their current parent_xform and child_xform API; no public attachment ID is required for the MVP.
  6. During finalize, the builder creates internal elastic endpoint cache records for every joint endpoint whose body is elastic.
  7. Builder validation ensures each elastic endpoint can sample mode values at the corresponding joint transform and that mode counts match the target elastic body.

This keeps the requested generalized layout while preserving VBD's current maximal-coordinate body solve. It also avoids forcing every physical joint on an elastic body to become a special elastic joint type.

VBD Solver Plan

Endpoint evaluator

Refactor the rigid VBD joint kernels around an endpoint function that can evaluate either a rigid endpoint or an elastic endpoint: 

Endpoint eval_joint_endpoint(joint, side, state):
    body = joint_parent or joint_child
    xform = joint_X_p[joint] or joint_X_c[joint]
    elastic = body_elastic_index[body]

    if elastic == -1:
        return rigid_body_anchor(body, xform)

    endpoint = joint_parent_elastic_endpoint[joint] or joint_child_elastic_endpoint[joint]
    eta = joint_q[elastic_q_start + 7 : elastic_q_start + 7 + N]
    x_local = xform.p + sum_i elastic_endpoint_phi[endpoint, i] * eta_i
    q_local = xform.q updated by elastic_endpoint_psi[endpoint, i] if supplied
    return world_endpoint_with_jacobians(body_q[body], x_local, q_local)

The evaluator should return world position/orientation, previous-step position, and block Jacobian columns for the active solve body. For a rigid body, the Jacobian has six columns. For an elastic body, it has 6 + N columns.

Elastic block descent

Add a separate elastic body VBD kernel, likely in newton/_src/solvers/vbd/reduced_elastic_kernels.py. For each elastic body, solve a small dense block: 

unknown delta = [delta_x(3), delta_theta(3), delta_eta(N)]

energy terms:
  inertial prediction of floating frame
  modal inertia
  modal stiffness: 0.5 * eta^T K eta
  modal damping / implicit velocity term
  body contacts using rigid proxy geometry at the floating frame
  joint constraints evaluated at rigid or elastic endpoints

solve:
  H * delta = -g
  apply line search / trust clamp compatible with VBD's current rigid step
  update body_q and eta
  normalize quaternion

Start with a model-level maximum mode count, for example 8 or 16, and store dense gradients/Hessians in flat scratch arrays. This fits Warp's GPU execution model without adding dependencies. Later, specialize kernels or use segmented dense block storage for larger reduced bases.

Joint support order

Stage Joint behavior Notes
1 JointType.ELASTIC owner only Zero constraints; verifies layout, FK sync, modal oscillator, and body pose copying.
2 Ball/fixed/revolute constraints to elastic endpoint positions Position constraints are the most important for a four-bar and provide finite-difference Jacobian coverage.
3 Revolute angular axis alignment with optional rotational modes If psi is not supplied, use the floating-frame orientation at the joint transform and document the limitation.
4 D6 and drive/limit projection Reuse existing D6 projectors after endpoint evaluation is generalized.
5 Deformed contact samples Out of MVP unless required. Start with rigid proxy shapes attached to the floating frame.

VBD control flow changes

  1. Initialization: copy the elastic owner joint's floating-frame pose into state_out.body_q and modal state into solver scratch.
  2. Coloring: include elastic bodies in the existing body conflict graph. Joints touching elastic endpoints create the same coloring conflicts as rigid joints.
  3. Iteration: run rigid body blocks and elastic body blocks by color. Elastic bodies use the 6 + N block solve.
  4. Finalization: finite-difference body_q to body_qd, update modal velocities, and write the full owner joint state back to state_out.joint_q/joint_qd.
  5. Controls: initially allow modal forces through the owner joint's modal DOFs in control.joint_f. Drives can be added later through the existing per-DOF target arrays.

The hard part is not adding the arrays. The hard part is making every constraint force/Hessian path use endpoint Jacobians instead of assuming a six-dimensional rigid endpoint. This should be the central refactor.

Generalized Solver Compatibility

Featherstone and MuJoCo paths should not be the initial target. Existing generalized solvers use joint-centric recursions and motion subspaces, while this VBD design treats the elastic owner as a body state block plus external constraint joints. Supporting generalized solvers later would require:

  • Extending jcalc_transform, jcalc_motion, and integration for JointType.ELASTIC.
  • Adding augmented spatial inertia and modal coupling terms to the articulated-body recursion.
  • Representing elastic endpoint constraints as loop constraints or using a different reduced-coordinate assembly.
  • Defining how physical parent joints and the floating-frame owner coexist in a tree articulation without double-owning one body.

For this feature request, the VBD implementation should be the source of truth and generalized solver support should raise a clear NotImplementedError until the reduced dynamics are mature.

Regression And Validation Tests

Per the repo guideline, implementation should start with a failing regression. The first test should demonstrate the requested state layout and fail before the new builder/model path exists.

Test Expected check Why
test_elastic_link_layout One elastic body creates one body slot and one owner joint with coord_count == 7 + N, dof_count == 6 + N. Locks the core state contract requested for joint_q/joint_qd and body_q/body_qd.
test_elastic_fk_body_sync eval_fk() copies owner floating-frame pose/velocity to the body slot and preserves modal coordinates. Protects render/collision compatibility.
test_elastic_joint_endpoint_jacobian_fd Analytical endpoint Jacobians match finite differences for translation, rotation, and modes. Protects VBD force/Hessian correctness.
test_vbd_single_mode_oscillator One free elastic body with one mode follows the implicit modal oscillator trend and damps energy. Validates the elastic block without joint complexity.
test_vbd_revolute_elastic_endpoint A rigid link connected to an elastic endpoint through a revolute joint keeps the anchor residual below tolerance. Validates the central connection requirement.
test_vbd_fourbar_elastic_coupler Loop closure remains bounded, modal amplitudes remain finite, and high stiffness approaches the rigid coupler result. Directly targets the user's four-bar example.
test_vbd_unsupported_solver_guard Non-VBD solvers fail clearly for JointType.ELASTIC until implemented. Prevents silent wrong physics.

Milestones

  1. Data contract and public API. Add JointType.ELASTIC, elastic mode and endpoint-cache model arrays, builder methods, validation, docstrings, and a failing then passing layout test.
  2. State synchronization. Add FK/IK or dedicated sync kernels for owner joint <-> body slot conversion. Ensure State.joint_coord_count and State.joint_dof_count remain correct with variable-size elastic joints.
  3. Endpoint abstraction. Refactor VBD rigid joint code so all joint constraints call a rigid/elastic endpoint evaluator. Add finite-difference Jacobian tests before broad solver work.
  4. Elastic VBD block. Implement 6 + N block solve for free elastic bodies with modal stiffness, damping, external modal forces, and final joint-array writeback.
  5. Joint constraints on elastic endpoints. Enable ball/fixed/revolute first, then D6. Build the elastic four-bar validation.
  6. Examples and docs. Add a minimal example in the existing Example class format with test_final(). Update public docs, solver support tables, changelog, and run docs/generate_api.py if public symbols are added.

Risks And Decisions

Risk Decision
FFRF inertia coupling can be oversimplified by a diagonal modal mass MVP. Store data so dense modal mass and inertia shape integrals can be added without breaking the API. Document the first solver's assumptions.
Attachment orientation is underdefined if only translational modes are supplied. Allow psi rotational/slope mode data. If absent, use floating-frame orientation and make angular elastic deformation unavailable for revolute axis alignment.
Existing JointType.dof_count() cannot express 6 + N with variable N. Add explicit per-joint coordinate/DOF sizing for elastic joints rather than overloading linear/angular axis count.
Four-bar loops may be modeled with equality constraints that VBD does not support. Use VBD-supported loop-closing joints for the elastic four-bar validation, not equality constraints.
Large mode counts could make each block solve expensive on GPU. Cap initial mode counts, preallocate flat dense scratch buffers, and treat higher counts as a later optimization problem.
